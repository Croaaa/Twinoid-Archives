An easy way to localize a piece of software for multiple languages is to put all the string values into a table, then call on that table each time a word is needed. Why? Because this gives the application excellent linguistic scalability. Every time you need a new language added, all you need to do is add in another "if" statement with the desired language and the words matching their respective strings.

However, this means that small mistakes can go unnoticed, because strings that aren't used often might have mistakes or incorrect values. When they get called upon, there's a chance that other strings might get called instead of throwing an error too. This happened to me recently when I was localizing the language for one of my projects; it simply said "%s" instead of putting the words there! I fixed it the following day, but that isn't always an option for software developers who miss a noncritical bug.

I'm not saying this is definitely what happened. But I am saying that the word "Undefined" is not supposed to be there, because it used to say "Missile" before.
