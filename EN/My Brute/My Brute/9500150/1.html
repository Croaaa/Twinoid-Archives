IMO it goes like this :
1) the character is born with a starting hash.
2) That hash is considered a Destiny, and is stocked in the database along with your character and remains unchanged. (unless you change the destiny by winning a tournament, then it generates a new hash)
3) THEN, at each level up, the character (and all his skills/belongings) is rehashed, and the resulting hash sent into a function that produces a new set of choice. (That resulting hash is temporary, and there is no need for it to be stocked)

Because the resulting hash will always be the same, and the function will always work the same way, there is no need to stock another data than the starting hash, it will on its own produce always the same result as long as you made the same decisions.

Also, maybe if the game is well made, certain competences have a something like score. Meaning, that if your character sucks, better options start to appear. But I doubt so.

I just demonstrated that it would be pointless.

New born phase :
===============
NAME:String = blablablabla (whatever the character name is)
ENDURANCE:integer = a (from 2 to 3 at the creation)
ARMOR:integer = b (from 2 to 3 at the creation)
STRENGTH:integer = c (from 2 to 3 at the creation)
AGITLITY:integer = d (from 2 to 3 at the creation)
SPEED:integer = e (from 2 to 3 at the creation)
STARTING HASH:string = aef12904887cde78 (I took the exemple of a MD5 checksum, 16 char. long)
WEAPONS:integer = 0 ( binary speaking, going from 0000 0000 0000 000 at the creation to 1111 1111 1111 1111 if the game had 16 weapons for example)
SKILLS:integer = 0 ( binary speaking, going from 0000 0000 0000 000 at the creation to 1111 1111 1111 1111 if the game had 16 skills)
PETS:integer = 0 to 31 (boolean 00000 to 11111, admitting of course that you could have the 5 pets at the same time)

Leveling up phase :
=================
This phase basically consists in generating a UNIQUE choice, which will ALWAYS remains the same, and which depends on your current character :

function getOption1(STARTING_HASH,NAME,ENDURANCE,ARMOR,STRENGTH,AGILITY,S PEED,WEAPONS,SKILLS,PETS) :return choice1
{ whatever
}

function getOption2(STARTING_HASH,choice1,NAME,ENDURANCE,ARMOR,STRENGTH,AGILITY,S PEED,WEAPONS,SKILLS,PETS) :return choice2
{ whatever
}

So if you observe how this is made, provided that nothing varies, you will ALWAYS have the same choices in output.
The ONLY 2 THING that could vary are :
- STARTING_HASH
- either ENDURANCE,ARMOR,STRENGTH,AGILITY,S PEED,WEAPONS,SKILLS,or PETS

BUT BECAUSE the brute name was also transmitted to the function, there will be NO WAY to reproduce the same exact INPUT situation with another brute.

Ergo, trying to detail a tree should be pointless, except for masturXXXXX. You WILL NEVER SEE the end of it because It is generated with an algorythm. (sorry to dissapoint)

PS : notice how choice1 is included as a parameter in the getOption2 function, This is because it includes also the possibility that choice2 is determined according to choice1.
