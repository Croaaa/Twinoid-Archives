Well, there's no reason that this can't be a completely user-run utility. Look at all of the map utility sites that have sprung up around AlphaBounce. At least one of them does indeed have a fan-coded submission program where players can send in screenshots of their levels for the map. It's not only NOT a crazy idea, but it's already been done for a Muxxu/Motion Twin game. I think you should go for it. I know that I'm already frustrated with the mapping system for Kube, and I'm sure that I'm not the only one. I'd personally most likely find the overworld map more useful than the "street view", but actual pictures of the dolmens would be a major boon in its own right.

All that being said, there is the not-so-small issue of this being a construction game, and thus this is not static stuff we're mapping. Any map would need to be updated at least daily to reflect all of the literal changes to geography that take place every time somebody plays. Unfortunately, this might require at least some interaction with the Kube developer team, if only to get some kind of script for taking current map snapshots directly from the server. And if they are upset about letting you have access to direct map data, you can let them know that the alternative is a computer controlled player that will wander the Kube world and refresh the map a few times a minute - potentially all day, every day, since it takes around a minute to cross a grid block. That's exponentially more server load than just fetching map pings from what they already use to produce their own in-game maps.

So, you have a heck of a challenge ahead of you, and good luck to you on it, but I believe that you can do it. Fans have gotten together to do extraordinary things before. Maybe this will be the next great chapter in Kube community evolution.

There's two obstacles: Kube's world is dinamic and people won't want to show their "secret areas".

So, my idea:

I'd suggest a simple map based on coloured squares(one square = one zone) based on Minemap(from AlphaBounce). It'd work like: green square for forest islands, orange for sequoia islands, yellow for desert and on, then gray squares for zones with very few kubes(island edges) or bridges, black square for never seen areas, light blue for zones where a player walked on without hoping inside a kube(swimming). Inside the squares, icons showing if the zone has a dolmen, if the zone was previously(or actually) rented, and if there's a screenshot(user must enter a link).

The purpose is that if a player want to go from X to Y, he can check if everything on the midway is grey or coloured to plan the route and if wanting to find for unexplored island need to go into the black areas. It doesn't need to be constantly updated. An entered data can only be removed by authorized people.

For adding data, a script that will get coordinates and see if the player is or not swimming.
In my opinion it could be very reliable and helpful. People could do severe bridge pillaging, but the map would store the other side of the bridge, so it can be rebuilt more efficiently. Also, asylums and hideouts would end up being exploited by explorators(which would go into black areas) with the time.

Just an idea.

Despite it's age, the french Kube community never made a real map.
I saw numerous projects start, and they all stalled because of the technical difficulties of Kube.

After many thoughts I have that to say:

While it would be nice to have a map accurate down to the kube, Morte745 is right to point that only generic per-zone information is really necessary.
A zone could be ocean, various qualities of bridge, any of the island types, or built/mess. Way easier to serve, and how often do you need more ?

I tried to make one by hand using online spreadsheets, but even such simplified data is way more they can handle.

A greasemonkey script can easely capture the coordinates and text in the side panel, and from that deduce portals (what durss's utility does), but also popular bridges and paths by collecting data from many players.
The color of the energy bar can hint at how much swimming is required along the way.

Player's in-game map could potentially be automatically captured as flash module's display can be read by utilities, but it is so cumbersome that people don't use it enough.

The radar is more interesting as it is always displayed and synchronised.
It spins so there is no easy kube-level capture, but is does no anti-aliasing besides the white dashes overlayed, so the main zone type (ocean / islands) on range can be deduced from the pixel colors.

Be it in-game map or radar, I have the full list of possible colors in Kube data, resource which was ported on xml.kubegb.fr.


Now, regarding the pictures... a greasemonkey script could extract picture information from users's profiles, so there is no major obstacles to a list of known pictures taken nearby a zone.
In fact, the biggest problem would not be collecting the data, but filtering an sorting it into something usable.


Of course, any form of automatic data collecting still requires users to install scripts and add-ons and browse pages, so in order for many people to do it this has to be bundled with something usefull.
I mean, the improved map or kube utility and the sniffer have to be the same thing, so that using it IS contributing to it.

And then, a secret mode is required as people don't want to give away their freshly discovered islands, bridges built or new bases.

Other features could be manual landmark entry, pathfinding, zone history, zone name & rental status, kind-of-kube searching, swimsuit-or-not branding of the bridges, searching/browsing of recent changes and so on.


Now that the hard part of the idea is done, we only someone to actually do the job.
If you ever get there let us know, the french Kube really need a map.
