coro

After taking a look at the code, you really should make this thing multilingual-friendly...

The general idea is to:

  • set up arrays of text strings, one per supported language
  • detect the domain that the script is active on, and copy the appropriate language array to a new variable
  • replace all instances of hardcoded text to be a reference to that variable

You can do this for both text strings and the links to tutorial threads (basically anything that would be different per server). There are 2 reasons to have a new variable be set, rather than just have one giant array to store everything. First, you can set the French array to be the "default" one, so the script won't have any errors or empty text strings. Second, you can have shorter references in the code (since you make a reference to a common variable, with no need to have a dynamic reference to the current language)

To combine arrays, use a line like this (variable names are highly inaccurate, this is just for illustration):

if (curLang = 'en') var finalLangArr = $.merge($.merge([], frenchArr), englishArr);

If you wish to use an object rather than an array, use this:

if (curLang = 'en') var finalLangArr = $.extend([], frenchArr, englishArr);

I only looked at the jQuery docs, I haven't tested anything out


Getting the script on Github will help get things done quicker, probably

 if(typeof(Main.k.currentLangpack[key]) != 'undefined'){

 var a = $.makeArray(arguments);
 a.shift();
 return vsprintf(Main.k.currentLangpack[key],a);
 }else{
 return Main.k.currentLangpack[key];
 }

 return '['+key+']';

Main.k.hasTranslation = function(key){
 if(typeof(Main.k.currentLangpack[key]) != 'undefined'){
 return true
 }
 return false;

Ah, I get what you're saying now. Basically you don't want to use screen/area identifiers for strings. I still think some kind of categorization is necessary. You're mentioning context but the context, while easy to determine and see in your own language, can get very tricky if you're re-using strings in other languages.

Simple example from German:

We (well, I) used the verb "Reload" for the button. In German I would most likely use "Neu Laden" or "Aktualisierien". Let's say there is a function to reload a gun somewhere (just bear with me ;-). In English it would work fine but in German you'd need to say "Nachladen" in this case. Whoever would make the code change for the new function would not know that, most likely, and just re-use the same string in the language he/she understands.

Generally speaking you can group strings by the following methods:
1. Screen/area (e.g. mainMenuEntry1, desertShopKeeper/forestShopKeeper)
2. The type of object they're connected with (button1Caption, questDescription1)
3. The function they serve or the action they trigger (I guess the closest we could get to the aforementioned "context"; reloadPage, displayText)

Usually some kind of mix of all three methods is used. The first method is definitely the one producing the most specific string-breakdowns and thus avoiding ambiguity but on the other hand makes minor changes to similar/duplicate strings across the code VERY time-consuming. I've seen strings auto-categorized by what code-module they were found in as well but this relies on the source code itself being well-designed and descriptive, otherwise you have string IDs that are easily traceable by the devs but completely useless for e.g. translators.

Anywho, how about we settle on mostly using the third categorization instead? Like, instead of "sidebarReload" and "sidebarReloadDesc" we'd have "reloadScript" and "reloadScriptDesc"? This would avoid misunderstandings if another type of reload was needed (like "reloadGun". In some cases it would indeed be possible to do without prefixes at all (like "Titles".
