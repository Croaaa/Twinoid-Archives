Database management is a mystery to me
But I'm assuming that skool set a seemingly low limit to encourage better code optimization (which would then lead to lower operating costs)

The 300 connection limit just refers to how many simultaneous database operations can occur
It's not like a hard limit on the number of people who can be logged into Twinoid
Most database operations take a fraction of a second to go through, but bad queries can result in an operation that just kinda grinds for a while and then stops after it takes too long

Also it's good to cache as much data as possible, so that the database(s) don't get accessed as frequently
Querying data from a database also takes longer than reading from the filesystem (even for an optimized query), so pulling data from cache files makes things load quicker

Also adding another database server involves some arcane voodoo magic
Either you have master and slave servers or you split up the data between different servers and somehow tie all of them together
I haven't had to deal with this stuff so I'm not too knowledgeable about it
