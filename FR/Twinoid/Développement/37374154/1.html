Bonjour à tous,

Je suis ici pour vous présenter un outil actuellement en développement, un framework pour les API Twinoid.

Twinoid fourni aux développeurs des API avec une documentation plus ou moins complètes, voilà pourquoi les développeurs ou futurs développeurs ne devraient pas être bridés et pouvoir développer leurs propres applications.



Fonctionnalités

• Gestion de session
Vous permet de garder en session l'authentification Twinoid (Le token).
• Gestion de l'authentification Twinoid
S'occupe de la redirection pour l'authentification Twinoid et la récupération du Token.
• Gestion de l'API Twinoid
S'occupe d'effectuer les requêtes à l'API Twinoid avec le token précédemment obtenu.
• Gestion de l'API Hordes
S'occupe d'effectuer les requêtes à l'API Hordes avec le token précédemment obtenu.
• Gestion de l'API Mush
S'occupe d'effectuer les requêtes à l'API Mush avec le token précédemment obtenu.
• Gestion des erreurs
Affichage des erreurs de requêtes.
• Calcul des appels vers l'API
Chaque API calcul le nombre de requêtes envoyées à l'API.
• Statut de Hordes sans requête supplémentaire
Il est possible de connaître le statut du jeu à la suite d'une requête précédemment effectuée sans requête supplémentaire. (Hordes)


Quel langage?
J’ai choisi PHP comme langage de programmation, il est simple d’apprentissage et supporté par tous les hébergeurs.

Ce framework est codé en POO, rien ne vous oblige à connaître celle-ci, il vous suffit de savoir utiliser ses objets avec les accesseurs fourni. Vous pourrez ensuite coder le reste en procédural si cela vous convient.

Si certains ont d’autres suggestions, je suis ouvert d’esprit.


Le concept
Ce framework est développé en POO et vous permet de gérer l’authentification avec Twinoid, avec les informations de votre application dans un fichier de configuration il vous permet de récupérer le token et de le stocker en Session ou comme bon vous semble.

Il vous suffit de créer une instance de la classe d’authentification et une fois le token récupéré il vous est possible d’instancier une connexion avec l’API Twinoid et/ou l’API Hordes. Pour l’instant je n’ai pas prévu gérer d’autre API, si vous avez des suggestions dites-les.

Ensuite, à partir de l’un de ces API, il est possible de récupérer les informations disponibles pour ceux-ci le tout dans des objets.

Concrètement, avec un appel vers
http://www.hordes.fr/tid/graph/me, vous recevez un objet User avec les mêmes informations que l’API. Ensuite, vous avez vos informations dans votre objet qui contient même un Array de Cadavers etc. Pour garder l'authentification, il y a plusieurs moyens comme de garder l'authentification en variable SESSION.



Exemple :
Visualiser avec coloration syntaxique.

<?php
 $hordesApi = new HordesAPI($authTwinoid->getToken());
 $currentUser = $hordesApi->getMe(); //Retourne un objet User (explication plus bas)
 echo $currentUser->name; //Affiche le pseudo de l'utilisateur
 // Il suffit de remplacer name par la valeur souhaitée de l'objet User.
 ?>

En somme, ce framework vous permet de développer vos propres applications sans vous soucier de l’authentification, la récupération des informations et les informations reçues sont déjà sous forme d’objets.

Le quota des applications est variable, tu devrais passer à 1000 après une première validation, car pour l'instant tu es le fier possesseur de la plus récente application .

Pour le système de cache, je n'avais pas prévu cela à la base, ça peut être intéressant dans certains cas comme pour les informations relatives à l'utilisateur qui ne devraient pas changer.

Par contre, il suffit de linéarisé l'objet retourné par une requête dans une variable session.

Par exemple pour les informations de l'utilisateur :

$_SESSION['currentUser'] = $hordesApi->getMe();
echo $_SESSION['currentUser']->name;
/* Affiche le nom sans faire une deuxième requête. */

Ainsi, à la prochaine itération de l'application, vérifié si la variable session ne contient pas déjà un objet, si oui l'utiliser, sinon le récupéré.

Ensuite, si cela devient essentiel à y penser rapidement je ferais des classes qui héritent des API et qui intègrent un système de cache. En bref, lors de la récupération des informations, l'objet vérifie avant s'il ne possède pas déjà ses informations et qu'ils datent de moins de x minutes, si oui il les retourne, sinon il récupère et retourne.

Certes, c'est comme ça que fait Fred pour rassembler les informations de tous les joueurs. Cependant le framework est là à la base pour récupérer les informations de l'API Twinoid, y intégrer une base de données ne serait pas une bonne idée.

Ensuite, dépendamment de l'application que tu fais, tu es libre d'utilisé une base de données. Pour des applications qui se concentrent sur l'utilisateur actuellement connecté et qui ne souhaite pas stocké ces informations dans une base de données, mais seulement analysé ceux-ci, à ce moment-là je conseille fortement de linéarité les objets reçus dans une variable session.

Concrètement, je reçois les informations d'un utilisateur x :
Linéarisation de ses informations dans une variable session (utilisable tant que l'utilisateur est connecté);
J'ajoute dans une autre variable session le timestamp d'expiration de ses données;
Lors de la prochaine itération de l'application, je vérifie s'il y a des données dans la variable session, si oui je vérifie si elles sont toujours valides (non expiré);
Dans le cas contraire, je récupère les nouvelles informations.

S'il s'agit simplement des méta-informations de l'utilisateur, pas besoin de gérer l'expiration de celles-ci.

C'est peut-être simple à première vue utilisé une base de données, cependant pour des applications simples, stockés les informations de chaque utilisateur qui visitent peut vite gourmand. De plus, pour récupérer les données sur l'utilisateur actuellement connecté dans la base de données, tu dois avoir l'identifiant unique de celui-ci, donc une requête à l'API, se qui revient au même que de le stocké en session. Le seul avantage que je vois à la base de données, c'est le fait qu'il est possible de garder un historique des informations et de faire des analyses.
