Bonsoir,

J'ai récemment eu des soucis avec le processus d'authentification à l'API Twino, et j'ai fini par les résoudre avec l'aide de Bugzilla.

Je vous propose donc une solution simple & efficace pour toutes vos APIs.

A faire
Créer une page session.php avec ce code : http://pastebin.com/tfbhrGwM et l'inclure au début de toutes vos pages php.
include_once(session.php);Dans session.php, pensez à modifier vos constantes d'application.


Créer une page redir.php avec ce code : http://pastebin.com/r4veXkfF et la définir comme URL de redirection de votre application.
Dans redir.php, n'oubliez pas de changer le path menant à votre fichier session.php.

Petites explication de redir.php :
La fonction get_twinoid_user_info() stocke dans la variable $_SESSION ces informations : uid, name, locale, avatar, oldNames, birthday.

La fonction get_info_on($scopePath, $fields) est un squelette qui est là pour vous aider à construire vos propres fonctions de récupération de données.
Les commentaires vous indiquent quoi faire dedans.

La fonction getting_work_done() est la fonction dans laquelle vous devez traiter toutes les informations que vous avez obtenues par les fonctions précédentes.
Faites chauffer vos DB

Merci encore à Bugzilla pour l'énorme aide sur ce code !

Il me semble qu'une petite explication historique serait bien utile...

L'exemple de code que j'ai mis sur GitHub et que j'ai fourni à Zeloo date à l'origine d'un test fait pour l’hébergement de scripts par l'Architectoire en 2013, donc il date d'avant le framework de Ralof.

Pour la petite histoire, il s'agit du site scripts.twinpedia.com, plus ou moins abandonné depuis 2014 vu que depuis lors, GitHub fait tout aussi bien l'affaire.

Ce framework de Ralof est très bien et j'avais envisagé à un moment de fusionner mon code avec le sien lorsqu'il l'a publié, surtout quand j'ai vu qu'il avait essayé de faire cela de manière propre et orientée objet. Mais j'ai vu ensuite que les principes utilisés étaient assez différents.

En particulier, pour le site de scripts et plus tard pour le site Fouille H24, je souhaitais que ces sites puissent être utilisés en étant connecté ou pas. Je voulais avoir un bouton "Connexion" / "Déconnexion" sur chaque page et permettre à l'utilisateur de choisir à n'importe quel moment d’être connecté au pas. Mon but était que l'utilisateur puisse s'identifier depuis n'importe quelle page et puis y retourner après connexion au lieu d’être redirigé vers la page d'accueil. C'était possible avec le code que j'avais écrit car j'encodais (de manière assez primitive, mais suffisante) l'URL de retour dans le champ "state" envoyé au serveur Twinoid, ce qui me permet de revenir sur la bonne page après décodage. Ce n'est pas possible avec le framework de Ralof qui utilise déjà le champ "state" pour indiquer l’état "connexion".

Ce serait quand même possible en utilisant le framework, à condition d'encoder l'URL de retour dans des données de la session PHP au lieu de le faire dans le champ "state" transmis au serveur. Ce serait peut-être même plus propre, mais il faudrait (ré)écrire ce nouveau code...

Une autre différence est que je voulais que les retours depuis le serveur Twinoid passent par une page intermédiaire (redir.php dans mon cas) qui me permettait de centraliser certaines informations avant de rediriger vers la page voulue. Cela me permettait entre autres de sauver certaines informations dans ma base de données pour ne pas devoir les récupérer à chaque fois. Ce n'est pas prévu dans le framework bien que cela puisse être ajouté facilement, mais vu que j'avais déjà écrit tout le code pour cela, j'ai préféré continuer à l'utiliser au lieu de le réécrire une fois que Ralof a publié son code.

En bref : le but n’était pas de dupliquer les efforts. Il s'agissait d'un projet parallèle commencé avant que le framework soit disponible. Vu que Ralof utilise une licence (MIT) plus permissive que celle que j'utilise (AGPLv3+), j'encourage ceux que cela intéresse à essayer ce framework.


Et j'oubliais une différence philosophique fondamentale qui rend le framework de Ralof difficilement utilisable pour moi : il utilise le CamelCase pour les noms de fichiers, ce qui heurte ma sensibilité et mes habitudes de vieux programmeur en C (je parle bien de C, pas de C++ ou C#) sous Unix/Linux.
