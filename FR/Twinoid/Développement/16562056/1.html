Suite à la sortie de l'application Mush Contagion on m'a demandé s'il existaient des tutos pour créer un langage comme celui créé pour cette application. Le langage mush, qui ressemble à ça :
▌▄◘▼▓ ▀ ▐▒▌ ▀▀▀

Si vous avez été infecté (et que l'appli est toujours active) ça se passe par là pour tester par vous même :
http://muxxu.com/a/mushcontagion/?act...

Pour être franc, je n'ai aucune idée de l’existence ou non de tutos et il existe très probablement des centaines de manières d'aborder le problème.

Mais voici une explication de ma manière de procéder.

Le cryptage :

  • 1 - La base, on récupère la chaine à crypter, disons "coucou".
  • 2 - Là, plein de solutions. L'idée est de convertir cette chaîne en binaire. Vous pouvez passer soit par les codes ascii de chaque caractères, soit, si votre langage le permet, écrire ça dans un tableau binaire (ByteArray). En flash j'utilise cette méthode très précisément.
  • 3 - Avant d'utiliser ces données binaires on va monter une bibliothèque de caractères. Pour cela suffit d'aller faire vos courses dans la table des caractères (sous windows, WINDOWS+R -> "charmap"). Pour le langage mush j'ai sélectionné 16 caractères. Ceci va permettre de faire coïncider 4 bits à 1 caractères.
    Avec 256 caractères on pourrait avoir 1 caractère pour 1 octet (8 bits). Concrètement, plus vous sélectionnez de caractères, plus l'encodage final sera court. Visez les puissance de 2 pour plus d'efficacité (2, 4, 8, 16, 32, 64, 128, 256, 512, 1024, etc...).
  • 4 - Notre bibliothèque de caractères et notre source binaire étant prêts il ne reste plus grand chose pour encrypter le message.
    Avec 16 caractères on sait qu'on peut remplacer 4 bits du message binaire par 1 caractère unique. On lit donc 4 bits des données binaires, on prend le caractère d'index correspondant, et on l'ajoute à notre message crypté. Concrètement ça donne quelque chose du genre dans une boucle :
  • var library = "▀▄█▌▐■▲►▼◄●◘◙▓▒░";
var bits = xxxx;
var char = library.charAt( bits );
message += char;

De cette manière on peut crypter n'importe quel message. Que ce soit un message déjà crypté ou non d'ailleurs.


Le décryptage?
Même chose mais dans l'autre sens
On lit chaque caractère crypté, on récupère son index décimal dans la librairie de caractères, et on l'écrit en binaire dans notre tableau de bits. Si un caractère n'existe pas dans notre librairie alors on l'ignore.
Ensuite on convertis cette chaîne binaire en chaîne de caractère, dans mon cas c'est géré par la méthode readUTFBytes. Mais en supposant que vous soyez partis sur les codes ascii des caractères pouvez arriver au même résultat à coups de fromCharCode par exemple.
 
 
Avec ça, vous avez une base pour encoder/décoder vos messages
Vous pouvez imaginer toute sorte de maltraitances sur les données binaires pour rendre l'algorithme plus sécurisé évidemment. Sans ça, à partir du moment où tous les caractères de la librairie sont découverts, il suffit de tester un décryptage avec toutes les possibilité d'organisation de ces caractères pour décoder vos messages. Un bon vieux brute force en somme.

Si tu veux juste te faire ton propre langage secret faut savoir que sans équivalent du ByteArray dont je parle dans mon exemple c'est un peu plus complexe.

Un moyen simple d'encrypter c'est ça par exemple :

var dictionary = ['╘','╙','╚','╛','╜','╝','╞','╟','╠','╡','╢','╣','╤','╥','╦','╧'];
var charToIndex = {};//Fast acces to a char index for decoding
for(var i=0;i<dictionary.length;i++) charToIndex[ dictionary[i] ] = i;

function encode(src) {
 var binary = '';
 //binary encoding
 for(var i = 0; i < src.length; i++) {
 binary += src.charCodeAt(i).toString(2);
 }
 
 var result = "";
 //replace every 4 bits by a char
 for(i = 0; i < binary.length; i+=4) {
 result += dictionary[ parseInt(binary.substring(i, i+4), 2) ];
 }
 return result;
}

encode('hello world'); //╥╙╡╟╞╞╤╥╧╘╟╟╥╧╤╣╞╞╜

Le problème c'est qu'en l'état tu ne pourras pas décoder cette chaine.
Pour décoder faudrait quelque chose du genre :

function decode(src) {
 var binary = '';
 //rebuild binary string
 for(var i = 0; i < src.length; i++) {
 binary += charToIndex[ src.charCodeAt(i) ].toString(2);
 }
 
 //???
}

Sauf qu'à la point d'exclamation il faut reconstruire la chaine à partir des données binaires. Mais tu ne peux pas savoir combien de bits il faut récupérer pour un caractère car ils n'auront pas tous la même taille.

Une solution "simple" pour ça c'est de s'assurer que tous les caractères soient encodés sur un nombre de bits minimum. 4 octets (32 bits) devraient être suffisants pour prendre en compte tous les caractères UTF-8.
En gros au lieu d'encoder la lettre "a" minuscule en "1100001" tu aurais "00000000000000000000000001100001".
Du coup, en partant d'un dictionnaire à 16 caractères comme dans mon exemple, un caractère "normal" serait représenté par 8 caractères "codés". C'est plus lourd.

Mais grâce à ça tu pourras décoder sans problème puisque tu sauras que 8 caractères n'en représentent qu'un.

Une autre solution est d'ajouter un caractère marqueur qui servira à séparer chaque caractère source.



L'avantage du ByteArray est qu'il fait tout ça de manière optimisée tout en proposant de compresser les données.
De mémoire pour mush-contagion je convertissait la chaine en binaire, comme dans l'exemple (sauf que c'était du vrai binaire puisqu'un ByteArray) puis je compressais ce ByteArray pour en réduire le poids. Ensuite je construisait le message codé en lisant le binaire compressé de la même manière que dans mon exemple au-dessus.
L'opération inverse se résumait à remplacer les lettres par leur équivalent binaire, à fourrer ça dans un ByteArray, décompresser le ByteArray, et récupérer la chaine décodée avec la méthode readUTFBytes() de la classe ByteArray.

Je le fais à partir d'une clé qui est la suivante :
f28bbade554294b95d4c370d0955b4495bce59b461113cd3999cb2dedb274a28b862187fbb061ad53f3343ba31fc89a89fbe6faa560f3bc0efd79019760602bc

L'intérêt du XOR c'est que tu effectue exactement la même opération pour crypter et décrypter.
Par exemple dans la console du navigateur tu peux essayer ça :

123456789^198122871 //ça donne 211079266
211079266^198122871 //ça donne 123456789

L'accent circonflexe c'est l'opérateur binaire XOR. A droite ça correspond en quelque sorte à la clé de cryptage et à gauche c'est les données sources à "crypter" (le texte converti en binaire dans le cas de mush contagion).

Pour faire ça sur une clé précise et sans avoir besoin que la clé fasse au moins la même longueur que le contenu à crypter (sinon seule une partie faisant la longueur de la clé sera cryptée) ça donne ça (c'est un c/c de la fonction que j'utilise de mon côté pour appliquer le XOR ) :

public function XOR(param1:ByteArray, param2:String):void{
  var i:int, len:Number, kLen:int, kIndex:int;
  len = param1.length;
  kLen = param2.length;
  for(i = 0; i < len; ++i) {
    param1[i] = param1[i] ^ param2.charCodeAt(kIndex);
    if(++ kIndex >= kLen) kIndex=0;
  }
}

Encore une fois j'utlise le ByteArray parce que plus simple et plus performant donc à toi d'adapter avec les outils que propose le langage que t'utilise .

Ben XOR avant compression c'est peu efficace et pour la sureté, et pour la compression.

La compression servira à rien, parce que, comme je suppose que ta clé est aléatoire, ton texte aura l'air bien plus aléatoire après xor qu'avant.
Donc la compression ne marche plus
si elle marche, c'est très mauvais signe : ça veut dire que ta clé est courte, et qu'on arrive donc à voir des redondances.
Normalement, c'est pas difficile à compter : en gros, les fréquences de lettres, c'est celles de la langue française. Donc 1 lettre sur 6 est un E, par exemple. D'où forcément intérêt de la coder avec une séquence courte (et, même raisonnement pour les séquences de lettres, je ne vais pas refaire l'algo LZW, mais bon, vous voyez : texte en clair en français=nombreux trucs fréquents).
Si tu xor avec un clé de 16 octets, ton E se retrouve réparti en 16 codes différents. Bien sûr, les autres lettres aussi vont partager certains de ces codes. Mais statistiquement, ça mélange déjà pas mal, ce qui fait qu'on voit beaucoup moins de séquences évidentes.

Si on en voit toutefois, c'est que ça va compresser un peu ; mais c'est aussi que ça commence à devenir possible de faire une analyse de fréquence sur ton texte xoré...

Et pire que le fait que la compression ne marche plus : ton xor sera plus prévisible. Parce qu'il suffit de décompresser, et se retrouver avec un xor sur un texte en clair, dans lequel on peut espérer (plus que sur un texte compressé) déduire la clé (méthode classique : autocorrélation pour deviner la longueur de la clé, une fois devinée la longueur de la clé, on a N sous-ensembles de ton texte sur lequel ton XOR n'est plus qu'un bête césar, cassable par une analyse de fréquence à la Al Kindi)



Au contraire si tu fais la compression avant, ton texte va bien se compresser. Et un texte bien compressé est humainement imbitable. Certes il y a une logique. Mais évidemment une analyse de fréquence marche bien sur du français parce qu'il y a des caractères plus fréquents que d'autres. Il n'y a aucune raison a priori de penser qu'un truc compressé contient des octets plus fréquemment que d'autres (au contraire : tout le jeu d'un bon algo de compression est de rééquilibrer la taille de l'encodage inversement proportionnellement à la fréquence ; ce qui fait que à la fin, ça étale très uniformément les probabilité d'occurrence de n'importe quel pattern).

Donc en plus d'être bien compressé, tu obtiens un truc pour lequel il va falloir se lever tôt avant de casser le xor par analyse de fréquence
(évidemment, il ne faut pas non plus être con et chiffrer l'entête du format de compression si tu utiliser un truc tout fait ; sinon ça revient à réléver d'entrée les x premiers octets de ta clé, x étant la taille de la "séquence magique" qui identifie ton format de compression)

L’étape de compression est réversible quand on connaît la méthode utilisée (deflate/inflate). Donc si on arrive à trouver la bonne table de caractères, l’étape de décompression est facile et il ne reste plus que l’étape du XOR, avec les défauts que l'on connaît pour les caractères qui se répètent.

Dans le cas 1 (XOR puis compression), le XOR rend le message un peu plus aléatoire que du texte pur et donc il se compresse moins bien. Le message aura donc une taille assez similaire avant et après compression, ce qui peut servir de guide pour celui qui ne dispose que du message codé et qui n'est pas certain de la transposition de la table de caractères : en faisant des essais, on peut généralement éliminer toutes les tentatives qui ne se décompressent pas du tout (erreur lors de la décompression) ou qui se décompressent avec une taille beaucoup trop différente du message codé. Lorsqu'on dispose de plusieurs messages codés et qu'on élimine les tentatives de décompression foireuses, on peut arriver à trouver la table de caractères et à décompresser tous les messages. Cela ne donne pas la clé XOR directement, mais à ce moment on peut comparer les messages et tenter de découvrir la clé. Si le message original est en français (ou une autre langue courante), on peut faire des suppositions sur la présence de lettres à tel ou tel endroit dans le message vu que les phrases commencent souvent par un article, qu'il y a des espaces entre les mots, que les voyelles reviennent souvent, etc.

Dans le cas 2 (compression puis XOR), le texte en clair se compresse relativement bien vu que le langage courant comporte un bon nombre de redondances. Le résultat de l’étape de compression donne quelque chose d'apparence aléatoire, qui sera encore un fouillis après application de la clé XOR. Vu que la compression ne laissera pas de séquences de bytes qui se répètent, il y a peu de danger que des morceaux de la clé XOR puissent apparaître facilement. Pour quelqu'un qui essaie de décoder et qui ne connaît pas la table de caractères utilisée, il ne sera pas facile de découvrir cette table de caractères car on ne peut pas faire de tentatives de décompression avec différents essais de tables de caractères vu qu'il faudrait à chaque fois avoir la clé XOR avant de décompresser. Et même en connaissant complètement la table de caractères, on ne peut pas retrouver la clé XOR facilement car on ne sait pas ce qu'on doit trouver : le message compressé a une apparence aléatoire et on ne peut pas faire de suppositions sur la présence de lettres à tel ou tel endroit.

Durss par aléatoire j'entends que tu n'as pas choisi comme clé la séquence utf8 correspondant au texte "la clé de durss", mais une séquence d'octets aléatoires, sans aucune logique (mais évidemment que tu gardes statiquement quelque part)

Sinon, si tu oublies la clé, comme dit plus haut, c'est totalement impossible de déchiffrer si la clé est de même longueur que le message (et que tu ne réutilises pas la clé pour plusieurs messages)

Et comme dit encore plus haut, si la clé ressert plusieurs fois (dans des messages différents, ou dans le même message parce qu'elle est plus courte que le message), c'est relativement la routine le déchiffrage :

  • l'auto correléation te permet de faire de bonnes hypothèses sur la taille de la clé (autocorrélation = gros mot, piqué des stats pour dire ici un truc assez simple : on décale le message de 1, et on compte le nombre d'octets qui restent inchangés. On fait pareil en décalant le message de 2. De 3, etc...
    Le décalage qui voit le plus d'octets inchangés a de fortes chances de correspondre à la longueur de la clé, ou en tous cas à un multiple de la longueur de la clé
  • Pour chaque hypothèse de longueur de clé, une analyse de fréquence sur le sous ensemble de message se fait comme pour déchiffrer un césar
    césar = on remplace chaque lettre par une autre ; analyse de fréquence = on compte combien de fois chaque code apparait. Si c'est un césar, alors l'un des codes correspond au E, et c'est probablement le code qui apparait le plus souvent, puisque c'est le E la lettre la plus fréquente. Ainsi de suite sur les autres lettres)
    Là, c'est un poil plus compliqué. Mais si l'autocorrélation t'a dit que la clé est de longueur 16, ça veut dire que tous les 16 octets, c'est le même xor qui est fait. Donc les octets 0, 16, 32, 48, 64, ... sont des octets correspondant à une bijection entre des lettres d'un texte français et un code. Donc cassable par analyse de fréquence (le code qui apparait le plus souvent est probablement un E).
    Idem pour les octets 1, 17, 33, 49, 65, ...
    Etc.
    Et à la fin, tu as ta clé.
    (C'est même plus facile qu'un césar, car dès que tu as trouvé le code qui correspond à E, tu as trouvé tous les autres codes pour les autres lettres. Vu que E^cle^E = cle. Du coup il suffit de xoré ton message avec E, et sur chaque sous ensemble de 1/16 du message, la clé est l'octet qui apparait le plus)

Evidemment il me faut pas mal de texte pour y arriver facilement. Mais bon, à cause du XOR (grâce à qui si je trouve le code d'une lettre je le trouve pour toutes), je pense que 30*N étant la longueur de la clé devrait suffire.
En dessous, il faudra tester plusieurs hypothèses. A 5*N tu commences à être sérieusement peinard (cad une clé dont la taille est 1/5 de la taille totale de tous les messages chiffrés). A N, tu as une garantie mathématique éternelle : même Dieu le père ne peut pas décoder ton message (autrement qu'en lisant dans ton cerveau).

Je ne connais pas spécifiquement "crypt" de php, mais vu son nom, c'est manifestement un portage de la fonction "crypt" standard d'UNIX (sinon PHP a l'habitude d'être plus verbeux sur les noms de fonction, quand ce ne sont pas que des portages de fonctions classiques).

Auquel cas, c'est le chiffrage DES, avec deux éléments : une clé, et un "sel".

Le "sel" c'est un truc bidon qui va transformer le message avant le chiffrage.
Le "sel" est public. Par exemple dans les vieux fichiers /etc/passwd d'unix, les 2 premiers octets du message chiffré, c'est le sel.
Il ne sert donc à rien strictement parlant en terme de chiffrage.

En gros si tu as une fonction de chiffrage

messagechiffre chiffre(message, cle);

ce que tu vas transmettre sur la place publique c'est

chiffre(message^sel, cle) . sel

(je parle dans un mélange de langages : "." c'est concaténation : tu affiches le résultat du chiffrage de "message xoré (par exemple) avec le sel", et tu affiches aussi le sel)

Quand tu voudras déchiffrer il te suffira donc de déchiffrer d'abord le messagechiffre (dechiffre(messagechiffre, cle)) ce qui te donnera comme résultat message^sel, et refaire un xor sel pour récupérer le vrai message.

Ça n'a pas d'intérêt flagrant : n'importe qui arrivant à casser ton chiffrage, récupérera "message^sel" et n'aura donc aucune difficulté supplémentaire à récupérer "message", puisque "sel" est public.

Le seul intérêt, c'est que justement ça rend plus difficile de casser le chiffrage. Car ça empêche l'analyse de fréquence entre messages.

Pour prendre un exemple concret, si on voit souvent le même message chiffré revenir, on en déduira que c'est un message banal.
Comme "Message reçu" par exemple.
Et ça donnera un indice pour déduire la clé.

Si à chaque fois qu'on chiffre un message, on y ajoute un sel aléatoire, ça devient plus difficile : chaque occurrence de "message reçu" va se trouver chiffrée d'une façon différente, et il sera donc impossible de se rendre compte qu'un message revient souvent.

Ce n'est pas un 2e chiffrage. C'est un truc qui sert à brouiller les pistes pour rendre plus compliqué le cassage du 1er (et seul) chiffrage.

Si tu chiffrais toujours la même chose avec la même clé, tu obtiendrais toujours le même résultat oui.

Mais ce n'est pas ce que tu fais. Ce que tu fais c'est chiffrer "message^sel" avec la même clé. Comme le sel change tout le temps, le résultat change tout le temps.
Et comme le sel est concaténé au résultat (les 2 premiers caractères du résultat), ça ne pose pas de problème que c'est "message^sel" qui a été chiffré, puisque tu peux retrouver "message" (tout ça est transparent pour l'utilisateur, évidemment. La fonction crypt fait les deux : elle passe le message au sel, et le chiffre ensuite)

Notes importantes : je dis "chiffrage / déchiffrage", parce que c'est ce que tu dis et c'est la question que tu pose ; et que cette technique du "sel" se retrouve aussi dans des fonctions de chiffrage déchiffrage.

Mais techniquement, si c'est bien la fonction crypt que je pense, c'est du hachage. Donc tu n'es pas censé pouvoir déchiffrer le message de toutes façons. Tu es seulement censé pouvoir vérifier le chiffrage.

En gros si je reprends l'exemple /etc/passwd, dans /etc/passwd tu trouves des

ayalti:Lpsdklfez83:/home/ayalti:...
nadawoo:UrIkdesldfskls:/home/nadawoo:...
durss:Ikslkdfsds:/home/durss


Lpsdklfez83 est le mot de passe "haché" de ayalti.
Et il a été haché avec une fonction de hachage après avoir été modifié avec du sel (qui est ici "Lp").

Le "sel" n'induite aucune difficulté supplémentaire de hachage. Pour vérifier un mot de passe à la connexion, il suffit de hacher le mot de passe saisi avec le même sel (Lp), et de vérifier qu'on obtient bien le même hachage.

L'intérêt du sel ici, c'est que si on est à 2 à avoir choisi comme mot de passe "toto", ça ne se verra pas directement (les mots de passes hachés ne seront pas identiques).
Sans le sel un vilain hacker peut piquer le fichier des mots de passe. Et s'il voit que sur la centaine de mots de passes il y en a 15 pareils (ce qui doit être réaliste), c'est que ces 15 personnes ont choisi des mots de passe très standards (chaîne vide, "toto", le nom de la boîte, etc). C'est donc ceux là qu'il doit essayer de casser.

Le principe est le même pour du chiffrage/déchiffrage. Mais à ma connaissance, crypt ne fait pas ça. Donc de toutes façons, tu ne peux pas déchiffrer.

Hop, confirmation rapide avec l'interprète php :

php > printf("%s\n", crypt("toto", "xx"));
xxFhi41qrWTno
php > printf("%s\n", crypt("toto", "xx"));
xxFhi41qrWTno
php > printf("%s\n", crypt("toto", "ii"));
iiGm7yjos6Ud.

Le sel est donc bien inclus dans le résultat (2 premiers caractères). Et quand on appelle 2 fois la fonction avec le même sel, on obtient le même résultat.

Ça c'est juste avec mon exemple "sel à 2 caractères". Quand on tape aucun sel, on a autre chose

php > printf("%s\n", crypt("toto"));
$1$S5A0XP3n$QLXgtWZr2RQ/CwXIsasIo1

Mais une rapide lecture de la donc montre que "$1$" c'est la signature d'un sel pour MD5 à 12 caractères. Donc je tente, je prends les 12 premiers caractères et je considère que c'est le sel

php > printf("%s\n", crypt("toto", '$1$S5A0XP3n$'));
$1$S5A0XP3n$QLXgtWZr2RQ/CwXIsasIo1

Bingo. C'était bien ça.

Donc, oui, le sel est inclus dans le résultat. Par défaut, apparemment, c'est du MD5, et le sel c'est les 12 premiers caractères (le défaut change peut-être suivant les installations de PHP. Mais la donc permet facilement de savoir lequel c'est. Et tu peux le changer via des variables globales).

Note : attention au ' et pas " dans mon deuxième appel (sinon php prend les "$" comme des préfixes de variables)
