Je crois qu'on s'emmêle un peu là.

Tout d'abord, MotionTwin produit des sites internets, il n'y a donc pas beaucoup de choix pour le code de description qu'on envoie à l'utilisateur, c'est à dire le code de base qui dis "là je veux ce message, par ici je veux ma topbar, rajoute moi une image en haut, etc.". Actuellement il y a deux langages pour mettre en place cette structure : xHTML et HTML5 (les deux sont très semblables au final ).
Pour son code de description, MT utilise xHTML 1.0 qu'elle étoffe avec une "feuille de style" (qui permet d'ajouter des effets de style, couleurs, soulignement, arrière-plan) écrite en CSS (comprendre Cascading Style Sheet).

Et donc ça c'est pour le code de description envoyé à l'ordinateur pour qu'il sache comment afficher la page, mais après il faut aussi utiliser un code pour le serveur, c'est à dire l'ordinateur qui envoie les pages. A quoi sers un tel code sur une machine qui ne fait qu'envoyer des pages ? Justement, c'est ce code qui permet de construire le code de contenu xHTML ! C'est le serveur qui décide quel contenu mettre au milieu de la page, tout en bas, qui récupère le pseudo et l'avatar des joueurs et les rapproche ensemble, c'est lui qui vérifie qu'il va bien t'envoyer les infos de ton compte, tes MP. Au final c'est vraiment le noyau solide d'un site web !
Donc quel code serveur est utilisé par MotionTwin ? C'est du PHP (comprendre ProProcessor Hyper... et après je sais plus )

Après il nous manque deux autres codes d'injonction (ou action), on peut même dire des "mini-programmes" : Flash et JavaScript. A quoi servent-ils si le serveur nous construit lui-même une page en xHTML qui, comme je l'ai dis, sers à afficher la page ? Et bien ils servent à faire plus que l'afficher, à la faire fonctionner.
C'est grâce à JavaScript par exemple que quand vous écrivez une lettre dans le champs de réponse du forum que la prévisualisation se crée, c'est aussi lui qui permet d'afficher une liste d'infos quand vous cliquez sur le pseudo, il permet aussi de faire fonctionner la recherche d'utilisateur (mais c'est un cas complexe). En fait JavaScript est un code qui permet de changer le xHTML (la présentation) sans rechargement de la page.
Flash quant à lui est très connu, car c'est lui qui fais marcher les jeux. Pour faire un vrai jeu à la façon MT il faut utiliser de très longs programmes en flash, celui-ci est une technologie qui permet de dire "affiche moi une image ici, déplace cet élément, rajoute de la musique, rajoute un effet de transition son et lumière et déplace cet élément en le faisant clignoter et en doublant sa taille, fais ça si je clique ici", enfin mes exemple sont plutôt nuls, mais c'est Flash qui gère les jeux.

Nous avons donc tous les langages utilisés par MT : PHP génère le contenu en xHTML, celui-ci est décoré par CSS et animé par JS alors que les jeux fonctionnent en Flash.

Alors maintenant revenons au cas particulier de l'HaXe, HaXe n'est pas un langage comme les autres... En fait Haxe est un outil pour programmer dans les autres langages. Chaque langage utilisé ci-dessus a ses propres règles, sa propre syntaxe. Il y a des choses qu'on peut faire on ne peut faire dans chaque langage et vu qu'il y en a beaucoup, ça devient vite long de coder dans tant de langages. Il faut donc voir HaXe comme un traducteur.

A titre d'exemple, afficher le message "Salut tout le monde !" en PHP s'écrit comme ça :
<?php echo "Salut tout le monde !";?>alors qu'en JS c'est :document.getElementsByTagName('body')[0] .innerHTML="Salut tout le monde !"; (ce ne sont que des exemples, il y a d'autres solutions.)
On voit bien que ce n'est pas du tout la même chose.

Et maintenant avec HaXe, il suffit d'écrire :trace("Salut tout le monde !");Ensuite on appuie juste sur un bouton et un programme traduit la syntaxe Haxe en JS, PHP, Flash, C++ et plein d'autres.

Le langage HaXe n'est donc pas utilisé à proprement parler pour les sites webs, mais plutôt pour leur création.
