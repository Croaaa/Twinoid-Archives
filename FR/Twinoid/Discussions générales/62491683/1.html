Quand les choses sont bien faites, une base de donnée ne stock que le mot de passe hashé. Quand tu remplis ton mdp dans le champ, le formulaire le hash de ton côté et envoi le hash vers le serveur pour comparé avec le hash dans la base de donnée. Si les deux hashs sont identiques, le mot de passe est le bon et donc ça valide l'ouverture de session. Au final, personne à par toi n'a jamais vu le mot de passe en clair.
Les adresses emails (ou les identifiants en général), elles, sont généralement stockées en claires, donc directement accessible et directement lisible côté base de donnée.

Ce que rajoute SSL, c'est que ce qui est transmit est hashé une nouvelle fois suivant la même procédure (plus ou moins). Ça rend l'identifiant illisible si tu surveilles ce qui se passe sur le réseau.

L'ID session est unique. Balance l'ID session à quelqu'un et il sera connecté avec ton compte. Sauf qu'il apparaît jamais en clair dans les URLs, donc à moins de faire de l'analyze un peu profonde de trafique / page web, tu auras jamais l'occasion de tomber dessus. Et quand bien même tu te fais avoir, Twinoid reset cet ID à chaque fois qu'il détecte un nouvel ordi. Ça t'oblige à te reconnecter si le cookie de connexion a sauté, sinon, ça réinitialise juste la connexion (e.g., tu vas finir sur la page d’accueil de Mush au lieu d'arriver directement en jeu et l'applet du jeu devra se relancer).

Pour les soucis où on est obligé de se reconnecter à chaque fois, je pense plus que c'est un soucis au niveau des cookies. Pas sûr de savoir pourquoi, mais ça a l'air extrêmement dépendant de l'user, donc je mettrais pas trop ça sur le dos de Motion Twin.

Quand tu remplis ton mdp dans le champ, le formulaire le hash de ton côté et envoi le hash vers le serveur pour comparé avec le hash dans la base de donnée.
Hmmmmm, ou pas. Si le hash est effectué côté client, ça veut dire qu'on peut lancer une requête de connexion post-hashage, donc directement avec le hash. Autrement dit s'il y a un leak de la base de données, la personne pourra se connecter sur n'importe quel compte puisqu'il aura directement le hash de la BDD lui même utilisé pour vérifier la connexion. Ainsi le hash est forcément appliqué côté serveur de tel sorte qu'une personne qui dispose du hash ne puisse pas se connecter avec, l'input du serveur n'étant pas le hash final mais justement le mot de passe, qu'il hash avant de comparer ce qu'il a en base.

Ou alors il faut un double hash. Le password subit un premier hash côté client, le hash étant lui même hash une seconde fois côté serveur, le serveur ayant en base de données le résultat du hash du hash. Oui là OK. Mais c'est d'abord un hash côté serveur, et facultativement un hash côté client. Si c'est ce que tu voulais dire on est d'accord.

Après le hash côté client ne change rien niveau sécurité sur le site donnée, il permet juste d'éviter que si la personne utilise le même mot de passe sur plusieurs sites, le "hackeur" qui espionnerait tes trames réseaux n'aura eu un visu qu'un hash qui sera inutile pour se connecter sur tes autres comptes. J'y avais jamais réfléchi mais c'est pas mal.
