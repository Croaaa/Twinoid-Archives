Avec "API" et "Multimedia" (toujours sans rien y connaître^^) j'excluerais quand même d'entrée de jeu .NET et les moteurs physiques.

.NET c'est un framework de programmation. C'est pas particulièrement orienté pour une tâche donné, et sauf erreur de ma part, ça doit forcément s'appuyer sur une API quelconque pour ce genre de chose (.NET ou pas, pour faire du son il faut une API son, pour faire de la 3D il faut une API 3D, etc).

Et les moteurs physiques, ben c'est des moteurs. Des librairies.
Je sais bien que beaucoup de gens confondent API et librairies mais ça n'est vraiment pas la même chose.
D'abord une API n'est pas une librairie (quand on a donné l'API, on n'a pas encore dit comme elle était fait. Pour parler terre à terre, l'API c'est le ".h" pas le ".c")
Et ensuite une API ne correspond pas forcément à une librarie (il y a des API de communication avec un système existant, par exemple. On appelle même API les trucs web, genre l'API Twinoid)

Donc qui dit API dit en réalité "norme", "protocole", "interface".
C'est une spécification de la façon dont on va discuter avec un truc.
Pour que j'appelle API un moteur physique (même en faisant abstraction du fait que l'API n'est que l'interface du moteur et pas le moteur lui-même - disons pour que j'appelle API le .h d'un moteur physique) il faudrait soit qu'il y ait plusieurs moteurs différents qui partagent la même API (auquel cas l'API fait office de barrière d'abstraction), soit que ce moteur tourne déjà (ce qui d'ailleurs est un peu la définition d'un moteur, mais c'est un autre débat : la plupart des "moteurs physiques" ne sont pas des "moteurs" mais de bêtes librairies de calcul) sur le système de toutes façons, et que l'API permet à des développeur d'aller interagir avec.

OpenGL est une API ; parce que quelle que soit la façon dont la 3D est implémentée sur le système, on sait qu'on peut l'utiliser de la même façon, en utilisant l'API OpenGL.
L'API Android est une API. Parce que le téléphone est là, existe, mais pourrait ne pas être programmable ; et l'est, parce que google fournit une API qui permet de d'interagir avec les composants logiciels et matériels du téléphone.

Un moteur physique c'est une simple librairie. Comme toute les librairies elle a une interface certes. Mais ce n'est pas une interface qui fait une barrière d'abstraction (il n'y a pas plusieurs moteurs physiques différents, partageant la même interface). Et ce n'est pas une interface qui permet d'accéder à un composant. C'est pas comme si tu avais de toutes façons un "moteur physique" qui tournait depuis toujours sur ton PC, mais que grâce au fait qu'on te fournit l'API, maintenant, tu peux le programmer : personne n'aurait jamais donné une librairie sans son interface de toutes façons.

Bref, grosse parenthèse (parce que la dernière fois que j'ai fait court sur le sujet, je me suis pris des remarques cinglantes des informaticiens du coin, pas conscients du fait que je savais un peu de quoi je parlais^^). Mais pour moi appeler API tes deux exemples serait une erreur dans le premier cas, et un gros abus de langage dans le 2e.


Quand je faisais appel à pinaillage, c'est plutôt pour le cas où quelqu'un aurait dit "depuis windows 7, il y a aussi la Microsoft Graphics Layer Interface ou je ne sais trop quoi". A dire vrai, je ne savais même pas que DirectX était encore utilisé (je pensais que ce n'était qu'une tentative désespérée de Microsoft de garder la main sur un des rares trucs qui échappe à leur monopole, comme NetBEUI ou VBscript)
